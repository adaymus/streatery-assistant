# v2 Architect Meeting Prep

**Goal:** establish the scope of parametric architectural drawing generation for v2 of the streatery pre-screener.

**Context:** v1 of the tool successfully geocodes a site, checks eligibility, computes a buildable envelope, and outputs a comprehensive submission package — except for the 5 architectural drawings (Site Plan, Elevations, Sections, Construction Details, Utility Access Plan), which currently appear as placeholders. v2 is about turning those placeholders into auto-generated drawings parameterized by site-specific data the tool already has.

**Reference:** DDOT Streatery Guidelines, FINAL, December 5, 2024 §5.2 — see also `CLAUDE.md` "Submission package requirements" for the full per-drawing content checklist.

Each section below has a few concrete decisions to resolve. A **recommended default** is noted for each — meeting can run agree-or-redirect rather than starting from a blank page.

---

## 1. Output format

| Question | Recommended default | Why |
|---|---|---|
| What file format does each drawing ship in? | **SVG generated by the tool → PDF rendered for TOPS upload.** Optionally export DXF/DWG from the SVG if the architect wants a CAD-editable copy. | TOPS only takes PDF. SVG is web-native and what we can generate programmatically. CAD format is for the architect's working files, not the deliverable. |
| Where do CAD source files live (if we maintain them)? | **The architect's repo, not ours.** Tool writes SVG; architect can import to AutoCAD/Revit if needed. | Separates concerns — we don't want to be the system of record for CAD edits. |
| Do drawings need title blocks matching a specific template? | **Yes, define a standard title block with District Bridges branding, drawing number (1-5), site address, date, drawing scale, "STRAWMAN — pending PE review" stamp until PE-stamped.** | Standardization across the cohort matters for PSC reviewers. |

## 2. Parametric inputs the tool already has per site

The tool's `PrescreenResult` carries all of this — confirming what's automatable vs what needs operator/architect input.

**Already in the data** (the architect can rely on these as parametric inputs):
- Envelope dimensions: length, width, position along blockface
- Blockface polyline geometry (WGS84 lat/lon, clipped to the building's block)
- Building lat/lon (from MAR)
- Side of street (Right/Left, mappable to N/S/E/W per orientation)
- Parking lane width per side (typically 8 ft on Mt Pleasant)
- Street speed limit + FHWA functional class → drives barrier type per §4.2
- Ward / ANC / SMD
- Cross-street names at each end of the block
- Every curb feature within range, each with lat/lon: parking meters (with METERID), fire hydrants, bus stops, bike lanes, street trees (with species + DBH + tree-box dimensions), ADA ramps, driveway curb cuts, crosswalks, loading zones

**Not in the data — needs operator/architect/site walk input:**
- Building footprint dimensions (we only have a point, not a polygon)
- Sidewalk width (real value varies — DDOT data we have doesn't include this directly)
- Fire Department Connection location on the building façade
- Subsurface utilities (Miss Utility / 811 call required)
- Aesthetic choices (materials, color, roof structure, lighting)
- Operator-provided dimensions (custom seating layout, table count)
- Frontage length per operator (currently defaults to 50 ft — UI input deferred)

| Question | Recommended default |
|---|---|
| Should v2 add a frontage-length UI input so the tool uses the real value per restaurant? | **Yes.** Already on the roadmap. Should ship before architect-drawing work to ensure the envelope matches what the architect will draw. |
| Should we attempt to derive building footprint from DC's address-point or building-footprint datasets? | **Attempt: yes for v2 if the data exists.** DC has a Buildings layer somewhere; brief investigation can confirm. If it's coverage-incomplete, fall back to "operator supplies dimensions." |
| Should we attempt to add sidewalk width from any DDOT source? | **Defer:** the Roadway Block layer has `SIDEWALK_OB_WIDTH` and `SIDEWALK_IB_WIDTH` fields — already pulled when geocoding but not currently surfaced. Easy to expose; worth doing for v2. |

## 3. Template library structure

| Question | Recommended default |
|---|---|
| One flexible parametric template, or N discrete templates (1-space, 2-space, 3-space+)? | **Hybrid:** one parametric base template with size variants. The base defines proportions, barrier placement, access ramp position; size variant fills in (1, 2, 3+ spaces of envelope length). Avoids combinatorial explosion. |
| What aesthetic palette? | **Utilitarian baseline + optional per-restaurant flourishes.** Concrete blocks or Jersey barriers per Guidelines §4.2, planters, simple wooden platform, removable shade structure. Restaurants can add branding via signage allowed by §4.7 (the 5.5"×8.5" signs we already generate). |
| Modular components vs monolithic design? | **Modular.** Barriers, platform, access ramp, overhead structure each as separate generatable components. Architect can swap one without redrawing the rest. |
| Different templates for Mt Pleasant cohort vs city-wide expansion? | **Start with one Mt Pleasant template.** Generalize later if District Bridges or another MSO opens a second corridor. |

## 4. PE-stamp workflow

Drawing 4 (Construction Details) requires a Professional Engineer's stamp per Guidelines §5.2 item 4. This is the most rigid constraint in the whole package.

| Question | Recommended default |
|---|---|
| One PE for the entire cohort, or per restaurant? | **One PE for the cohort.** Massive cost savings if the same parametric design is used; PE stamps the design once and re-stamps per-site only if site-specific details differ. |
| Wet stamp or digital? | **Confirm with the PE (DC allows both; preferences vary).** Digital is dramatically faster for re-stamping per site. |
| Does the PE review each generated drawing before submission, or only the template? | **Decision:** ideally the PE reviews the template once + spot-checks instances. The tool should flag any instance where site-specific dimensions differ enough that re-review is warranted (e.g., barrier type changes due to functional class). |
| What constitutes a "material change" that requires re-stamp? | **Open question for the PE.** Likely: different barrier type, envelope >2× template size, atypical foundation. |

## 5. Automation scope by drawing

| # | Drawing | Automatable today? | Architect-required scope |
|---|---|---|---|
| 1 | Site Plan | **High** — all curb feature data + envelope geometry already available | Define standard symbols + title block conventions |
| 2 | Elevations | **Medium** — envelope dimensions + barrier type known; aesthetic details not | Architect defines elevation template per template variant |
| 3 | Sections | **Medium** — parking lane width + ADA path geometry derivable; access ramp + platform construction not | Architect defines section conventions (where to cut, what to show) |
| 4 | Construction Details | **Low** — needs structural design, materials, fasteners; PE-stamped | Architect + PE design; tool generates the cover page + site-specific dimensions only |
| 5 | Utility Access Plan | **Medium** — above-ground features known; subsurface utilities require Miss Utility call per site | Architect defines symbology; tool generates point locations from existing curb feature data |

| Question | Recommended default |
|---|---|
| What's the v2 MVP — all 5 drawings or just 1-2? | **Site Plan + Utility Access Plan first.** They're the most data-backed and lowest architect-effort. Ship those, demo to PSC reviewers, get feedback, then tackle 2/3/4. |
| When does the tool hand off to the architect? | **After SVG generation.** Tool produces the SVG; architect imports to their CAD tool of choice, refines, exports PDF, uploads to TOPS. Future: direct PDF generation when refinements stabilize. |

## 6. Cohort coordination (Block Permit vs Design Permit)

District Bridges files **one** Streatery Block Permit covering the corridor; each restaurant files its own Streatery Design Permit. Per Guidelines §5.2, the Block Permit requires its own Site Plan + Curbside Management and Delivery Plan.

| Question | Recommended default |
|---|---|
| Should the tool generate the Block Permit's Site Plan as a corridor-wide composite? | **Yes for v2.** This is uniquely valuable — only the tool has the data to overlay every cohort restaurant's envelope on a single corridor map. |
| Who drafts the Curbside Management and Delivery Plan narrative? | **District Bridges + architect collaboration.** Tool provides the data table (existing meters, loading zones, etc.) and a narrative template; the operator/architect fills in the specifics about reroutes. |
| Should the cohort plan auto-update when a new restaurant joins? | **Yes.** Re-run, regenerate. Saves manual work as the cohort expands. |

## 7. Review + approval workflow

```
Tool generates SVG
  → Architect imports + refines in CAD
    → Operator reviews
      → PE seals Drawing 4 (Construction Details)
        → All 5 drawings uploaded to TOPS as PDFs
          → PSC review (Design Permit) OR DDOT admin review
            → Public notice posted 10+ days before PSC hearing
              → Approval → Build → Inspections (DDOT + DOB)
                → Certificate of Occupancy + Use → Open
```

| Question | Recommended default |
|---|---|
| Where do revisions happen — back in the tool or directly in the architect's CAD? | **CAD for v2 instances; the tool's parametric template for systemic improvements.** A revision that applies to every site goes upstream; per-site revisions stay in the architect's files. |
| Site walk: before generation, after generation, or both? | **Before** — the pre-screener's site-walk checklist (already generated) covers the unknowns. Architect should not start drawing until the walk is done. |
| Who maintains the parametric template long-term? | **Open question.** Could be the architect (with the tool calling out to their template repo), District Bridges (with the architect contracting in), or the tool author. |

## 8. Deliverable timeline

| Milestone | Estimate (architect involvement TBD) |
|---|---|
| Site Plan SVG generation, 80% complete for cohort templates | 2-3 weeks of dev + ~10 hrs architect time |
| Site Plan + Utility Access Plan generation, both shippable to TOPS | +1-2 weeks |
| All 5 drawings generated to architect-review quality | +3-4 weeks + PE engagement |
| Cohort-wide Block Permit Site Plan | +1 week (reuses single-site infrastructure) |
| Production-ready v2: end-to-end TOPS submission from one button click | Open — depends on how much architect refinement turns out to be required vs automatable |

## Decisions to bring out of the meeting

By end of meeting, ideally agree on:

1. **Output format**: SVG → PDF for TOPS. Maybe DXF for the architect.
2. **v2 MVP scope**: Site Plan + Utility Access Plan only, or all 5?
3. **Template approach**: parametric base + size variants vs N discrete designs.
4. **PE engagement**: who, when, wet or digital stamp.
5. **Source-of-truth ownership**: who owns the parametric template, who owns per-site instances.
6. **Frontage-length UI input**: ship before drawing work or in parallel?
7. **Building footprint data source**: investigate DC Buildings dataset or punt to operator input?
8. **Sidewalk width**: surface from already-fetched Roadway Block fields (quick win).
9. **Open question fallbacks**: what to do when Miss Utility hasn't been called yet, when sidewalk width is unknown, etc.

## Appendices

### Pre-screener data the architect can pull per site

Every pre-screen exports a structured JSON result (in the submission package's appendix, or via the CLI `npm run prescreen -- "<address>"`). Key fields the architect's parametric template can reference:

```jsonc
{
  "geocoded": {
    "mar": { "fullAddress", "latitude", "longitude", "streetNumber" },
    "addressPoint": { "blockKey", "routeId" },
    "block": {
      "blockName", "fromStreet", "toStreet",
      "speedLimitMph", "functionalClassFhwa", "functionalClassDc",
      "parkingLaneWidthPerSideFt", "totalParkingLanes",
      "wardId", "ancId"
    },
    "side": "Right" | "Left",
    "blockface": {
      "blockfaceKey",
      "geometry": { "paths": [[[lon, lat], ...]] },  // WGS84 polyline
      "combinedMeasFrom", "combinedMeasTo"             // route measures in meters
    }
  },
  "eligibility": {
    "verdict": "ELIGIBLE" | "ELIGIBLE_WITH_CAVEATS" | "INELIGIBLE",
    "envelope": {
      "lengthFt", "widthFt",
      "approximateParkingSpaces", "recommendedTemplate",
      "geometry": { /* GeoJSON LineString */ },
      "startAlongBlockfaceFt", "endAlongBlockfaceFt"
    },
    "bindingConstraints": [...],
    "extensionOpportunity": { "couldHelp", "extendedEnvelopeLengthFt" }
  },
  "curbFeatures": {
    "parkingMeters": [ { "location": { "latitude", "longitude" }, "metadata": { "meterId", "spaces", ... } }, ... ],
    "fireHydrants": [ ... ],
    "busStops": [ ... ],
    "bicycleLanes": [ ... ],
    "streetTrees": [ ... ],          // includes species, DBH, tree-box dimensions
    "adaCurbRamps": [ ... ],
    "driveways": [ ... ],
    "crosswalks": [ ... ],
    "loadingZones": [ ... ]
  }
}
```
